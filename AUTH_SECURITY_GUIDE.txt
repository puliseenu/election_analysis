===============================================
DATABASE STRUCTURE
===============================================

auth_database.db (SQLite)

USERS TABLE:
  • id (Primary Key)
  • username (Unique)
  • email (Unique)
  • password_hash (PBKDF2-SHA256, 100k iterations)
  • role: 'admin' or 'user'
  • status: 'pending', 'approved', 'rejected'
  • created_at: registration timestamp
  • approved_at: approval timestamp
  • approved_by: admin username who approved
  • last_login: last login timestamp
  • is_active: 1 (active) or 0 (deactivated)

AUDIT_LOG TABLE:
  • Tracks all actions: login, registration, approval, etc.
  • Records username, action type, timestamp, details
  • For compliance and debugging

SESSIONS TABLE:
  • Stores active sessions with tokens
  • Tracks login IP, user agent
  • Session expiration management

===============================================
SECURITY FEATURES
===============================================

PASSWORD SECURITY:
  • Passwords hashed with PBKDF2-SHA256
  • 100,000 iterations + random salt
  • Never stored in plain text
  • Cannot be reversed

ACCOUNT STATUS:
  • pending: Registered but not approved
  • approved: Can login
  • rejected: Cannot login (permanent)

ROLE-BASED ACCESS:
  • Admin: Full access + user management
  • User: Dashboard access only (when approved)

AUDIT TRAIL:
  • Every action logged: login, registration, approval
  • Timestamps and user info recorded
  • Useful for security investigation

===============================================
TROUBLESHOOTING
===============================================

Q: "Password or username incorrect"
A: • Check caps lock
  • Verify username exists
  • Ensure account is approved (not pending)

Q: "Username already exists"
A: • Choose different username
  • Or try forgot password (contact admin)

Q: "Email already registered"
A: • That email is already in use
  • Contact admin if it's your email

Q: "Account pending approval"
A: • Your registration is waiting for admin
  • Admin will review and approve
  • Check back later

Q: "Database locked"
A: • Multiple apps accessing database
  • Stop other Python processes
  • Restart dashboard

Q: Lost admin password
A: • Delete auth_database.db
  • Restart dashboard
  • Admin account will be recreated (admin/admin1432)

===============================================
SECURITY BEST PRACTICES
===============================================

✓ DO:
  • Change admin password regularly
  • Use strong passwords (8+ chars, mixed case, numbers)
  • Keep Python/dependencies updated
  • Review audit logs periodically
  • Backup auth_database.db regularly

✗ DON'T:
  • Share admin credentials
  • Use simple passwords (123456, password, etc.)
  • Leave dashboard running on public internet unsecured
  • Store auth_database.db in cloud without encryption
  • Reset auth_database.db in production

===============================================
PRODUCTION DEPLOYMENT
===============================================

When deploying to production:

1. CHANGE ADMIN PASSWORD:
   • Delete auth_database.db
   • Restart dashboard
   • Admin account recreated
   • Change password immediately

2. USE STRONG PASSWORDS:
   • Admin: Complex 16+ character password
   • Document securely

3. DATABASE BACKUP:
   • Backup auth_database.db regularly
   • Store backups securely
   • Test restore procedures

4. ENABLE HTTPS:
   • Use reverse proxy (nginx)
   • Install SSL certificate
   • Redirect HTTP to HTTPS

5. FIREWALL/NETWORK:
   • Restrict dashboard access by IP
   • Use VPN for external access
   • Don't expose port 8050 directly

6. MONITORING:
   • Check audit logs regularly
   • Monitor failed login attempts
   • Set up alerts for unauthorized access

7. RATE LIMITING:
   • Implement login attempt limits
   • Prevent brute force attacks
   • Add account lockout after N failures
